    线程池

    线程池的本质是预先启动一批线程，然后从某种任务队列里面选择一个任务给一个特定的线程去执行。 线程池具体如何设计，需要做到什么程度，与具体业务目标需求以及任务粒度有很大的关系。 同时，必须考虑下面几个关键性质，但是这三个不能兼得。 1，吞吐量（单位时间内任务完成总数） 2，高user time CPU利用率（业务逻辑所占的CPU利用率） 3，公平性（各任务之间的公平性） 因此，把实现留给业务实现者，是更合理的选择。

    wait：若进入wait状态，解锁互斥锁。若被唤醒，重新上锁互斥锁。

    std::condition_variable的成员函数notify_one()与notify_all()是用来唤醒阻塞在wait（）的线程。 假如有多个线程调用condition_variable：：wait（）陷入休眠之后。condition_variable的实现中有一个等待队列来保存堵塞在它之上的线程。 当其他线程调用notify_one()时，只唤醒等待队列中的第一个线程；其余的线程不会被唤醒，需要等待再次调用notify_one()或者notify_all()才会唤醒。 如果是调用notify_all()：会唤醒所有等待队列中阻塞的线程，但是存在锁争用，只有一个线程能够获得锁。队列中的其他线程，会不断尝试获得锁，当第一个唤醒的线程释放锁之后，剩余的线程就能获得锁继续执行。

    原文链接：https://blog.csdn.net/qq_34999565/article/details/120876201 第一种形式只有一个参数unique_lock&，调用wait时，若参数互斥量lck被锁定，则wait会阻塞。

    第二种形式除了unique_lock&参数外，第二个参数pred，即函数指针。当函数运行到该wait()函数时， 带条件的被阻塞：wait函数设置了谓词(Predicate)，只有当pred条件为false时调用该wait函数才会阻塞 // 竞争条件锁，pred为false 才会阻塞。pred为true，才会解除阻塞。 // 启动线程池，m_bShutDown = false;此时 m_TaskQueue 空，pred 为 false， 阻塞线程 // 关闭线程池，m_bShutDown = true;此时一定不wait。 m_condition_var.wait(lock, [this] {return m_bShutDown || !m_TaskQueue.IsEmpty(); });

    m_cv.wait(lock, [this] {return !m_queue.empty(); }); m_cv： m_cv是一个std::condition_variable对象，用于线程间的同步。它允许一个线程在某个条件不满足时进入等待状态，并在条件满足时被唤醒。 lock： lock是一个std::unique_lock对象，用于保护共享资源（这里是消息队列m_queue）。std::unique_lock比std::lock_guard更灵活，因为它可以被解锁和重新锁定。 [this] {return !m_queue.empty(); }： 这是一个lambda表达式，捕获了当前对象的上下文（this指针）。它的作用是检查消息队列m_queue是否为空。 如果m_queue不为空（!m_queue.empty()返回true），则条件满足，线程继续执行。 如果m_queue为空（!m_queue.empty()返回false），则条件不满足，线程进入等待状态。

    何时激活线程： 对于第一种形式void wait (unique_lock& lck); 只要其它线程调用notify_one()或者notify_all()函数，wait()就会结束阻塞。 对于第二种形式，除了需要调用其它线程调用notify_one()或者notify_all()函数外，还需要pred()为true才会解除阻塞。 join()函数：注意join后的线程运行结果后就会回收资源，子线程消亡。 当使用join()函数时，主调线程阻塞，等待被调线程终止，然后主调线程回收被调线程资源，并继续运行； detach()函数： 当使用detach()函数时，主调线程继续运行，被调线程驻留后台运行，主调线程无法再取得该被调线程的控制权。 当主调线程结束时，由运行时库负责清理与被调线程相关的资源。

    future和promise

    future\promise的含义：

    future表示一个可能还没有实际完成的异步任务的结果，针对这个结果, 可以添加回调函数以便在任务执行成功或失败后做出对应的操作； promise交由任务执行者使用，任务执行者通过promise可以标记任务完成或者失败； 交互流程: 在父进程中: 定义promise, 通过promise.get_future() 获取对应的future, 在父进程中将promise传递给子进程；然后进行非阻塞地干自己(父进程)的事情, 在子进程中通过 promise->set_value,为共享状态设置处理后的值; 然后在父进程中通过future.get()获取共享状态设置处理后的值 参考：https://blog.csdn.net/shyjhyp11/article/details/135754168

    future状态

    在并发编程中，我们通常会用到一组非阻塞的模型：promise\future。在python、js、java中都提供future\promise，是现代语言常用的非阻塞编程模型。 参考示例：https://zhuanlan.zhihu.com/p/672327290 std::future 在 C++ 中表示一个异步操作的结果，它可以处于以下几种状态：

    未初始化（Uninitialized）: std::future 对象刚被创建，但还没有与任何异步操作关联。此时调用 .get() 将抛出 future_error 异常，错误码为 future_errc::no_state。

    等待（Pending）: std::future 对象已经与一个异步操作关联，但是该操作还没有完成。调用 .get() 会导致调用线程阻塞，直到异步操作完成。

    就绪（Ready）: 异步操作已经完成，结果已经可用。此时调用 .get() 将立即返回结果，而不会阻塞。

    已获取（Retrieved）: std::future 对象的结果已经被获取一次或多次。如果 std::future 是通过 std::shared_future 共享的，那么结果可以被多个 std::shared_future 对象获取。

    已丢弃（Discarded）: 如果 std::future 对象的结果已经就绪，但是 std::future 对象被销毁了，那么结果就被丢弃了。这通常发生在 std::future 对象超出作用域并且没有被 .get() 调用的情况下。

    已取消（Deferred）: 当使用 std::async 启动异步任务时，如果指定了 std::launch::deferred 策略，那么异步操作将被推迟执行，直到 .get() 被调用。此时 std::future 对象处于一种特殊的状态，它既不等待也不就绪。

    请注意，std::future 对象本身没有直接的方法来查询它的状态。状态的判断通常是通过尝试调用 .get() 并捕获可能抛出的异常来间接实现的。 例如，如果 .get() 抛出 future_errc::no_state 异常，则表示 std::future 对象处于未初始化状态； 如果调用 .get() 导致线程阻塞，则表示它处于等待状态； 如果 .get() 立即返回结果，则表示它处于就绪状态。

    future.wait_for((std::chrono::milliseconds(100)) 进入阻塞状态。超出参数时间或者异步操作的结果已经准备好，会停止阻塞。

    std::Foward

    std::forward是C++11中引入的一个函数模板，用于实现完美转发。它的作用是根据传入的参数，决定将参数以左值引用还是右值引用的方式进行转发。 传统上，当一个左值传递给一个函数时，参数会以左值引用的方式进行传递；当一个右值传递给一个函数时，参数会以右值引用的方式进行传递。 完美转发是为了解决传递参数时的临时对象（右值）被强制转换为左值的问题。std::forward实现完美转发主要用于以下场景：提高模板函数参数传递过程的转发效率。
