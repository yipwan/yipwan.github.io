    MVVM基础

    MVVM 特性

    • Model（数据模型）
    • View（界面）
    • View-Model（视图的抽象，即“没有界面”的 UI 逻辑）

    三者的分工一句话： View 只负责“长什么样”；Model 只负责“数据长什么样”；ViewModel 负责“界面该有什么数据、能做什么动作”，并借助“绑定引擎”把两头的变化自动同步。

    View-Model：行为

    // ViewModel/MainViewModel.cs
using System;
using System.ComponentModel;
using System.Runtime.CompilerServices;
using System.Windows.Input;          // ICommand
using DemoMvvm.Model;

namespace DemoMvvm.ViewModel;

public class MainViewModel : INotifyPropertyChanged
{
    private readonly Person _person = new();   // 聚合 Model

    // 把 Model 的属性“包装”成 View 需要的格式
    public string Name
    {
        get => _person.Name;
        set { _person.Name = value; OnPropertyChanged(); }
    }

    public int Age
    {
        get => _person.Age;
        set { _person.Age = value; OnPropertyChanged(); }
    }

    // 计算属性，也会通知
    public string Summary => $"{Name} is {Age} years old.";

    // 按钮能点的命令
    public ICommand BirthdayCommand { get; }

    public MainViewModel()
    {
        BirthdayCommand = new RelayCommand(() => ++Age); // 点一下长一岁
    }

    // ---------- INotifyPropertyChanged ----------
    public event PropertyChangedEventHandler? PropertyChanged;
    protected void OnPropertyChanged([CallerMemberName] string? p = null)
        => PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(p));
}

// 最简 ICommand 实现，不写第三方包
public class RelayCommand : ICommand
{
    private readonly Action _exec;
    public RelayCommand(Action exec) => _exec = exec;
    public bool CanExecute(object? _) => true;
    public void Execute(object? _) => _exec();
    public event EventHandler? CanExecuteChanged { add { } remove { } }
}

    CommunityToolKit作用

    CommunityToolkit.Mvvm 是微软官方维护的一个开源库（属 .NET Community Toolkit 系列），它把 MVVM 架构里最常用、最样板化的代码封装成“源代码生成器”——编译时帮你生成属性、命令、消息通知等代码。 作用一句话：写更少的代码，实现完整、高性能、可测试的 MVVM 模式。

    传统MVVM

    // ViewModel
public class CounterViewModel : INotifyPropertyChanged
{
    private int _count;
    public int Count
    {
        get => _count;
        set { _count = value; OnPropertyChanged(); }
    }

    private ICommand _addCommand;
    public ICommand AddCommand =>
        _addCommand ??= new RelayCommand(() => Count++);

    public event PropertyChangedEventHandler PropertyChanged;
    protected void OnPropertyChanged([CallerMemberName] string p = null)
        => PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(p));
}

    ToolKit提供的类型

    observableobject

    简化了数据绑定的编写。

    OnPropertyChanged作用是当属性值发生改变时，OnPropertyChanged 负责 发出“我变了”的通知，让 UI 层（数据绑定引擎）立即重新读取该属性，从而刷新界面。

    使用observableobject则自动进行数据刷新。

    RelayCommand

        // 1. 继承 ObservableObject 即可
    public partial class CounterViewModel : ObservableObject
    {
        // 2. 标记 [ObservableProperty] 即可
        [ObservableProperty]
        private int count;

        // 3. 标记 [RelayCommand] 即可 
        [RelayCommand]
        private void Add() => Count++;
    }

    添加[RelayCommand]特性后，会自动生成AddCommand（会在函数名后加Command）方法。

    “属性包装字段” 本身非常常用，但 “自己先手写一个私有字段，再手写一个属性去包它” 只有在需要加逻辑时才这么做。[ObservableProperty] 会自动生成一个首字母大写的公开属性，并把它做成可通知的完整实现。

    添加特性。例如NotifyCanExecuteChangeFor。

    将一个按钮命令绑定到AddCommand函数上。

    <Page …
      xmlns:local="using:Demo.ViewModels">
    <Page.DataContext>
        <local:MainViewModel x:Name="Vm"/>
    </Page.DataContext>

    <StackPanel>
        <!-- 1. 输入框：双向绑定 Count 属性 -->
        <TextBox Text="{x:Bind Vm.Count, Mode=TwoWay, UpdateSourceTrigger=PropertyChanged}"
                 InputScope="Number"/>

        <!-- 2. 按钮：Command 绑定 -->
        <Button Content="添加"
                Command="{x:Bind Vm.AddCommand}"/>

        <!-- 3. 结果 -->
        <TextBlock Text="{x:Bind Vm.Total}"/>
    </StackPanel>
</Page>

    private void Add()
    {
        Total += Count;
    }

    private bool CanAdd() => Count > 0;          // ← 真正的判断逻辑

    [ObservableProperty]
    private int _total;
}

    [NotifyCanExecuteChangedFor] 就是 “属性一变，帮我顺便告诉某某命令重新判断自己还能不能执行” 的省力语法糖，如果不能执行则直接置灰。

    Massage机制

    解决多个view-model之间相互通信。

    本质是让一个类统一对所有的view-model的消息时间进行统筹规划。成为事件聚合者，Event Aggregator。是多个publisher和多个subscritor之间的桥梁。和中介者模式比较像。

    Event Aggregator的功能

    1） 全局唯一，任何地方都可以使用。可以设置为单例。

    2）具有Register方法，任何view-model注册后，都能收到Send的消息

    3）具有Send方法，任何view-model注册后，都能发送消息
