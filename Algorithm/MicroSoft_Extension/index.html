    借助App.config文件管理.NET应用程序的配置项（传统配置方法）

    常见配置方式

    1) app.config (xml格式)

    2) appsetting.json(json格式，只读，不可写)

    3) config.ini (ini格式) 一般不用

    任何一个c#项目只要包含App.config，编译后会产生 .dll.config 文件。

    app.config 作用

    在 .NET 项目里，app.config（对于 Web 项目则是 web.config）是 CLR 最优先读取的应用程序配置文件。 它的作用一句话概括： “让同一份程序在不同环境、不同机器、不同用户下都能用不同的外部参数运行，而无需重新编译。”

    app.config 特性

    1. 物理文件： 开发阶段叫 App.config（位于项目根目录）。 编译后自动复制到输出目录并改名：xxx.exe.config（xxx 是你的程序集名称）。

    (WPFFramework是exe.config，WPFCores是dll.config，因为一部分被拆分到dll种）

    1. 本质： 一份符合 CLR 规范的 XML 文件，运行时 System.Configuration 会把它映射成强类型对象供代码调用。

    2. 读取时机： AppDomain 启动时由 CLR 一次性加载，之后只读（若要改需另写代码）。

    app.config 配置项

    最常用appSetting和connectionStrings。

    节点 用途 典型场景
    <appSettings> 自由键/值对 数据库连接串、功能开关、路径
    <connectionStrings> 专放连接串 EF、ADO.NET 用
    <startup> CLR 版本、位数 强制用 .NET 4.x、prefer 32-bit
    <runtime> 程序集绑定、GC 设置 重定向 DLL 版本、关闭并发 GC
    <system.serviceModel> WCF 端点、绑定 服务端/客户端配置
    <log4net>、<nlog> 第三方配置 日志等级、输出目标
    自定义 Section 强类型业务参数 邮件服务器、报表模板目录

    一份appSettings举例

    <?xml version="1.0" encoding="utf-8" ?>
<configuration>
  <!-- 1. 自定义开关 -->
  <appSettings>
    <add key="DebugMode" value="true"/>
    <add key="ReportFolder" value="D:\Reports"/>
  </appSettings>

  <!-- 2. 数据库连接 -->
  <connectionStrings>
    <add name="Default" connectionString="Server=.;Database=ShopDB;Trusted_Connection=True"/>
  </connectionStrings>

  <!-- 3. 强制使用 .NET 4.8 -->
  <startup>
    <supportedRuntime version="v4.0" sku=".NETFramework,Version=v4.8"/>
  </startup>
</configuration>

    代码中如何读取app.config

    ConfigurationManager.AppSettings["Flag"]

    代码中如何写app.config

    var cfg = ConfigurationManager.OpenExeConfiguration(ConfigurationUserLevel.None)
var flag = cfg.AppSettings.Settings["Flag"].Value
cfg.AppSettings.Settings["Flag"].Value = 2
cfg.save()

    ​ 再打开exe.config或者dll.config，此时值已经被修改。

    用appsettings.json文件管理配置项（先进配置方法）

    基本用法

    安装nupkg包：Microsoft.Extension.Configuration.Json

    多配置文件

    optional ：该配置文件是可选的

    reloadOnChange：文件修改的时候，重新读取

    多文件可son以多次使用AddJsonFile() 读取多个文件

    多文件同名字段读取规则：按照AddJsonFile顺序进行覆盖。例如appsettings.json 文件和appsettings.development.json文件。

    作用：

    1、多环境部署。生产环境、测试环境有不同的配置，按照优先级顺序进行读取

    2、安全性。不需要把多个配置文件推送到git上。

    注意：

    该json文件只读，不允许修改后再次save。

    将配置映射到配置类上

    安装Microsoft.Extension.Configuration.Binder

    class UserSetting
{
    public string? Title {get;set}
    public int Width {get;set}
    public int Height {get;set}
}

var UserSettings = configuration.GetSection("UserSettings").Get<UserSettings>();// 一个secion表示配置文件中的一个配置

    注意：Get到类中后，reloadonChange 不生效，需要重新Get到类中

    IoC

    基本概念

    Inversion of Control 控制反转

    依赖：一个对象需要另一个对象来完成某个功能

    用于实现控制反转的管理器

    .Net 中常用MicroSoft.Extensions.DependencyInjection包

    IoC作用

    将依赖注册的好处：1）控制生命周期 2）自动装配，免于传参 3）一个接口可能有不同实现

    通俗来说，

    • 用 IoC 是因为 “不想让业务代码里到处都是 new，从而换实现、写测试、管生命周期都只需要改启动时那一小块地图。”
    好处 没有 IoC 的场景 用了 IoC 后的场景
    可测试 new SqlRepository() 写死，单元测试必须连真数据库 注入 MockRepository，测试毫秒级
    可替换 换缓存实现要改 10 个类 改一行注册 RedisCache→ICache
    解耦 业务代码 #include "sql.h" 业务代码只认识接口
    生命周期集中管理 到处 static 手写单例，线程安全自己管 容器统一给单例/作用域
    配置即代码 编译期决定实现 启动时按配置决定注入哪个实现（功能开关、A/B）
    横切关注点 日志、事务、监控散在业务里 用装饰器或动态代理统一注入

    IoC实现

    典型的建造者模式。

    var builder = new ServiceCollector(); // 实现控制反转的容器
builder.AddTransient<ICounter, Counter>() // 添加瞬态生命注册，每次GetService都新建一个对象
builder.AddTransient<ILogger, MyLogger>()
builder.AddTransient<CounterService>()
//builder.AddSingleton<ICounter, Counter>() // 添加单例生命注册，每次GetService都新建一个对象
var services = builder.BuildServiceProvider();
var logger = services.GetService<ILogger>();

class CounterService
{
    private ICounter _counter;
    private ILogger _logger;
}

Interface ICounter
{
    void Increase();
    void Decrease();
}

class Counter:ICounter
{
    void Increase();
    void Decrease();
}

Interface ILogger
{
    void Log();
}

class Logger:ILogger
{
    void Log();
}

    注意：如果ICounter 有多个实现，则getservice得到最后一个注册的实现类

    思考：直接new 一个CounterService，为什么要兜一圈用ServiceCollector进行注册？

    带参数注册

    如果CounterService的初始化函数比较复杂，注册的时候给具体的实现
builder.AddTransient<CounterService>(sp => new CounterService
                                     (
                                        sp.GetService<ICounter>(),
                                        sp.GetService<ILogger>(),
                                        1   
                                     ))

    WPF实战

    安装CommunityToolKit.MVVM

    StartupUri作为WPF的入口标记

    xaml中默认使用StartupUri=“MainWindow.xaml” 作为入口.StartupUri 就是 WPF 的 “入口标记”： 指定哪一页/哪个窗口在程序启动时自动打开。

    WPF使用IoC注册

    1）xaml中删除StartupUri=“MainWindow.xaml”

    2）在App.xaml.cs 中重写 OnStartup函数

    public partial class App : Application
{
    public static IServiceProvider Services { get; private set; }

    private void OnStartup(object sender, StartupEventArgs e)
    {
        var serviceCollection = new ServiceCollection();
        ConfigureServices(serviceCollection);
        Services = serviceCollection.BuildServiceProvider();

        // 从容器拿窗体
        var mainWindow = Services.GetRequiredService<MainWindow>();
        mainWindow.Show();
    }

    private void ConfigureServices(IServiceCollection sc)
    {
        // 注册界面
        sc.AddSingleton<MainWindow>();
        // 注册业务
        sc.AddSingleton<ICounterService, CounterService>();
    }
}

    根据配置注册不同的模块

    appSettings.json

    {
  "Data": {
    "Repository": "Sql"   // 或 "Redis"/"Mongo"
  }
}

    对应的POCO

    public class DataOptions
{
    public string Repository { get; set; } = "Sql";
}

    条件注册

    var builder = WebApplication.CreateBuilder(args);

// 1️⃣ 把配置绑到 POCO
builder.Services.Configure<DataOptions>(
    builder.Configuration.GetSection("Data"));

// 2️⃣ 读配置
var repoKind = builder.Configuration["Data:Repository"];

// 3️⃣ 条件注册
switch (repoKind)
{
    case "Sql":
        builder.Services.AddScoped<IUserRepo, SqlUserRepo>();
        break;
    case "Redis":
        builder.Services.AddScoped<IUserRepo, RedisUserRepo>();
        break;
    case "Mongo":
        builder.Services.AddScoped<IUserRepo, MongoUserRepo>();
        break;
    default:
        throw new InvalidOperationException($"Unknown Repository:{repoKind}");
}

builder.Services.AddControllers();
var app = builder.Build();
app.MapControllers();
app.Run();

    或者不想 switch：把“名字→类型”也写在配置里，用反射一次搞定

    {
  "Data": {
    "Repository": "Sql",
    "TypeMap": {
      "Sql":   "MyApp.SqlUserRepo, MyApp",
      "Redis": "MyApp.RedisUserRepo, MyApp"
    }
  }
}

    var typeMap = builder.Configuration
                     .GetSection("Data:TypeMap")
                     .Get<Dictionary<string,string>>(); // {"Sql":"...,..."}
var key = builder.Configuration["Data:Repository"]!;
var implType = Type.GetType(typeMap[key], throwOnError: true)!;
var serviceType = typeof(IUserRepo);
builder.Services.AddScoped(serviceType, implType);

    C++中实现IoC的方法

    方案 依赖 反射 单例/生命周期 适合场景
    宏工厂 0 × 手动 嵌入式、脚本绑定
    模板 mini 容器 C++11 / boost::any × 可扩展 中小项目、学习
    Boost.DI 仅头文件 × 完整 商业/大型工程

    宏实现

    #include <iostream>
#include <string>
#include <functional>
#include <unordered_map>

// 1. 全局工厂表
using Creator = std::function<void*()>;
inline std::unordered_map<std::string, Creator>& gRegistry() {
    static std::unordered_map<std::string, Creator> inst;
    return inst;
}

// 2. 注册宏
#define REG_CLASS(cls)                                \
    namespace {                                       \
        struct _Reg_##cls {                           \
            _Reg_##cls() {                            \
                gRegistry()[#cls] = []()->void*{      \
                    return static_cast<void*>(new cls()); \
                };                                    \
            }                                         \
        } _instance_##cls;                            \
    }

// 3. 获取宏
template<class T>
T* Get(const std::string& name) {
    auto it = gRegistry().find(name);
    return it == gRegistry().end() ? nullptr
                                   : static_cast<T*>(it->second());
}

// 4. 业务代码
class ILogger { public: virtual void log() = 0; };
class ConsoleLogger : public ILogger {
public:
    void log() override { std::cout << "console log\n"; }
};
REG_CLASS(ConsoleLogger)   // 编译期塞进表

int main() {
    ILogger* p = Get<ILogger>("ConsoleLogger");
    if (p) { p->log(); delete p; }
}

    模板方法

    #pragma once
#include <functional>
#include <unordered_map>
#include <boost/any.hpp>
#include <memory>

class IocContainer {
    using Creator = boost::any;
    std::unordered_map<std::string, Creator> map_;
public:
    template<class T, class... Args>
    void registerType(const std::string& key) {
        auto ctor = [](Args... args) -> T* {
            return new T(args...);
        };
        std::function<T*(Args...)> f = ctor;
        map_[key] = f;
    }

    template<class T, class... Args>
    std::unique_ptr<T> resolve(const std::string& key, Args&&... args) {
        auto it = map_.find(key);
        if (it == map_.end()) return nullptr;

        // 取出擦除后的 function 并调用
        using Func = std::function<T*(Args...)>;
        Func creator = boost::any_cast<Func>(it->second);
        return std::unique_ptr<T>(creator(std::forward<Args>(args)...));
    }
};

    使用

    #include "IocContainer.hpp"
#include <iostream>

struct IRepo { virtual int get() = 0; };
struct SqlRepo : IRepo {
    int get() override { return 42; }
};

struct Service {
    IRepo* repo;
    explicit Service(IRepo* r) : repo(r) {}
    void work() { std::cout << "data=" << repo->get() << '\n'; }
};

int main() {
    IocContainer ioc;
    ioc.registerType<SqlRepo>("SqlRepo");
    ioc.registerType<Service, IRepo*>("Service");

    auto repo = ioc.resolve<SqlRepo>("SqlRepo");
    auto svc  = ioc.resolve<Service>("Service", repo.get());
    svc->work();
}

    Boost.DI

    #include <boost/di.hpp>
#include <iostream>

namespace di = boost::di;

struct IRepo { virtual int get() = 0; };
struct MockRepo : IRepo {
    int get() override { return 99; }
};

struct Service {
    IRepo& repo;                 // 依赖
    Service(IRepo& r) : repo(r) {}
    void run() { std::cout << repo.get() << '\n'; }
};

int main() {
    // 1. 创建容器并绑定接口→实现
    auto injector = di::make_injector(
        di::bind<IRepo>.to<MockRepo>()   // 告诉 DI：谁要 IRepo 就给 MockRepo
    );

    // 2. 一键解析整条依赖链
    auto* svc = injector.create<std::unique_ptr<Service>>();
    svc->run();
}
